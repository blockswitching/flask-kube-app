How It All Works Together

From code to container — the full picture

The Big Picture

This project demonstrates how a Python web application goes from source code on your machine to a portable, runnable container. Here's the flow:

1

Write the App

Create your Flask application in app.py with routes, templates, and static files.

2

Define Dependencies

List all Python packages in requirements.txt so they can be installed in the container.

3

Write the Dockerfile

Create step-by-step instructions for building the Docker image — base image, dependencies, code copy, and start command.

4

Build the Image

Run docker build -t flask-kube-app . to create a portable image containing your app and all its dependencies.

5

Run the Container

Run docker run -p 5001:5001 flask-kube-app to start a container from the image and map port 5001.

Why This Approach?

Reproducibility

The container runs the same way on your laptop, your colleague's machine, or a cloud server. No more "works on my machine" problems.

Isolation

Your app runs in its own environment. Different projects can use different Python versions without conflicts.

Deployment Ready

The same image you test locally can be deployed to Kubernetes, AWS ECS, Azure App Service, or any container platform.

What Gunicorn Does

In development, Flask has a built-in server (app.run()). But for production, we use Gunicorn — a production-grade WSGI server that:

  • Handles multiple requests concurrently with worker processes
  • Is more stable and performant than Flask's dev server
  • Manages worker lifecycle (restarts crashed workers)
# This is what our Dockerfile uses to start the app:
CMD ["gunicorn", "--bind", "0.0.0.0:5001", "app:app"]

# "app:app" means: in the file "app.py", use the variable "app"
